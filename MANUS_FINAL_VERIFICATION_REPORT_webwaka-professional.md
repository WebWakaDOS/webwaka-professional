## Issues Found and Resolved

| # | Issue | Severity | Root Cause | Fix |
|---|-------|----------|------------|-----|
| ISSUE-1 | RBAC context injection failure — banner endpoint returned `403` instead of `404`/`400` for authorized `TENANT_ADMIN` role | **High** | `jwtAuthMiddleware` from `@webwaka/core` sets `c.get('user')` and `c.get('tenantId')` but NOT `c.get('role')` or `c.get('userId')`. All three module APIs (`event-management`, `legal-practice`, `matters`) read `c.get('role')` directly, getting `undefined`, causing every RBAC check to fail with 403 | Added a second `app.use('/api/*')` middleware after `professionalAuthMiddleware` in all three modules to extract `role` and `userId` from the `user` object into context |
| ISSUE-2 | `wrangler.toml` KV namespace IDs invalid — `TENANT_CONFIG` and `EVENTS` bindings had UUID-format IDs (`d870d125-...`, `d1d3b537-...`) that do not exist in Cloudflare | **Critical** (blocks deploy) | Replit Agent used placeholder UUIDs instead of real Cloudflare KV IDs | Created 4 new KV namespaces (`webwaka-professional-tenant-config-staging/prod`, `webwaka-professional-events-staging/prod`) and updated `wrangler.toml` with real IDs |
| ISSUE-3 | CI deploy jobs used GitHub `environment:` context (`name: staging`, `name: production`) which blocks access to repo-level secrets, causing `wrangler-action` auth error 10000 | **Critical** (blocks CI deploy) | Same pattern as webwaka-ui-builder and webwaka-ai-platform | Removed `environment:` context from `deploy-staging` and `deploy-production` jobs; replaced `cloudflare/wrangler-action@v3` with `npx wrangler deploy` using explicit `CLOUDFLARE_API_TOKEN` env vars |
